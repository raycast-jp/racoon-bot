# Design Doc

技術設計書（Design Doc）を書く・レビューするためのスキル。Google などで使われる「議論のための設計ドキュメント」スタイルに沿う。要点だけを書き、ソースコードとペアで最大効果を出すことを意識する。

参考: https://zenn.dev/shimakaze_soft/scraps/7a8170927a30cd

## いつ書くか（最初に判定する）

書く前に必ず「そもそも書くべきか」を判断する。

- **書かない**: プログラマーが自分だけ **かつ** 実装が 2〜3 日以内で終わる規模。
- **書く**: 複数人が関わる / 実装に 4 日以上かかる / 設計の選択肢が複数あり後で理由が分からなくなりそう / 後から入るメンバーがコード理解に時間がかかりそう。
- **タイミング**: 要求仕様が一通り決まった後、実装フェーズの前。執筆に 1〜2 週間以上かかるなら、対象範囲を分割する。

ユーザーの依頼がこの「書かない」条件に当てはまりそうなら、Design Doc ではなく短い Design Proposal（後述）や箇条書きメモで十分かもしれない、と提案する。

## 書き方の原則（ドラフト中ずっと守る）

1. **What / Why / How を書く**: 何を・何のために・どのように。特に「決定に至った理由・背景」を重視する。決定事項の羅列にしない。
2. **要点だけ。書き過ぎない**: ソースコードを見た方が早い情報（細かい I/F 仕様、自明な実装詳細）は省く。「読んで分かるが無駄がない」を目指す。
3. **ソースコードとペア**: コードだけでは伝わらない粒度の大きい情報（設計意図、他モジュール／システムとの関係、トレードオフ）を書く。
4. **長文にしない**: 項目を区切り、箇条書きと表を活用してテンポ良く。Detailed Design は数ページに収める。長くなるなら機能単位で別 Doc に分割する。
5. **図は文章の補助**: 構成図・クラス図・シーケンス図・状態遷移図を添える。ただし図やイラストだけにしない（後から意味が読み取れなくなる）。自然言語の説明を必ず添える。
6. **生きたドキュメント**: 試行錯誤の途中アイデアでも書き出し、何度も直しながら仕上げる。Design Doc は「思考の道具」でもある。
7. **完璧を目指さない／時間をかけすぎない**: 早い段階でレビューに出す。実装しないと分からない点は早めに実装で確かめる。

## テンプレート構成

`templates/design-doc-template.md` をベースに使う。必須セクションとオプションセクションは以下。

### 必須（ほぼ必須）
- **タイトル / 著者 / 日付 / ステータス**: 名前は良いものを付ける（設計がはかどる）。
- **Goal（目的・ゴール）**: 1〜3 文で「このシステムの目的は★であり、◯◯を実現するのがゴール」と明確に。曖昧だと設計が迷走する。PRD があればリンク。
- **Non-Goals（やらないこと）**: 「××機能は含まない」と除外を明記し、議論の範囲を狭める。Goal が広く誤解されると議論が脱線する。
- **Background（背景）**: なぜ作るのか（動機・経緯）。前提知識・参考リンク・専門用語/略語の定義。関係者全員が理解できるよう補足する。
- **Overview / High-Level Architecture（概要・全体構成）**: 「全体から部分へ」。3〜4 段落＋システム構成図で俯瞰を示す。
- **Detailed Design（詳細設計）**: 各パートを項目分けし、**データ構造 / アルゴリズム / インタフェース** を端的に。必ず「なぜその設計にしたか（設計判断の理由）」を添える。疑似コードやチャートは必要なら。
  - 項目分けの方針: 機能で分ける（小・中規模で多い）か、モジュール・画面など物理配置で分ける。
- **Alternatives Considered（検討した代替案）**: 採用しなかった設計を、その利点・欠点（トレードオフ）込みで書く。見落とされがちだが重要。
- **Test Plan（テスト方針）**: テスト観点と方法。テスト規模の見積りも前倒しできる。
- **Caveats / Security / Privacy（注意点・セキュリティ・プライバシー）**: 懸念は専用章で議論を明確化し、ローンチ前に必ず解決する。

### オプション（必要なものだけ）
- プロジェクトメンバー / レビュワー（Who）
- 要求仕様（PRD）へのリンク・要点
- 既存システムとの相違点 / 関連システム
- 各パートの実装場所（ファイル名等）
- 使用例（サンプルコード）
- API セクション（既存 API 公開時。完全な I/F 定義はコピペせず、トレードオフに関わる部分に焦点）
- データストレージ（いつ・どんな形で保存が発生するか。スキーマ全文ではなくトレードオフ中心）
- 既知の問題
- 運用方法 / モニタリングプラン
- 参考文献 / この Doc の格納場所 / メモ / 変更履歴

## 執筆プロセス

1. **ドラフト**: Goal / Overview / Detailed Design を先に書く。未確定部分は `TODO:` で残す。
2. **最初のフィードバック**: 近いメンバーやメンターに共有し反映。
3. **関係者・チームレビュー**: 必要に応じ広く共有。チーム内完結なら省略可。
4. **開発**: レビュー後に実装。設計変更が出たら、緊急時を除き Doc を直してからコードを直す。
5. **メンテナンス**: 改修・新メンバー加入・大変更時に更新する。

## 格納・運用

- バージョン管理できる場所に置く（Markdown を Git リポジトリに、が基本。コードと同時レビューしやすい）。
- エンジニア以外のコメントが重要なら Google Docs も選択肢（同時編集・コメントが容易、ただし履歴は追いにくい）。
- コードレビュー時の必須資料にする運用が望ましい（レビュワーはまず Doc を読み、それからコードを読む）。

## 代替フォーマット（Design Doc が過剰なとき）

- **Design Proposal**: Design Doc の簡略版。Detailed Design 以下を省き、気軽にフィードバックをもらう用。
- **PRD（Product Requirements Document）**: 「何を作るか／作らないか」を議論したいときはこちら。Design Doc より前（または同時）に書く。
- **コードレビュー**: API 細部・アルゴリズム実装の議論は、実装後のコードレビューの方が具体的で建設的。

## 進め方（このスキルを使うときの手順）

1. **対象を理解する**: 要求仕様・関連コード・関係者を確認。情報が足りなければユーザーに質問する（Goal、対象範囲、制約、既存システムとの関係など）。
2. **書くべきか確認**: 「いつ書くか」の基準に照らし、過剰なら軽量フォーマットを提案。
3. **テンプレを埋める**: `templates/design-doc-template.md` をコピーし、必須セクションから埋める。不明点は `TODO:` で明示。
4. **原則でセルフレビュー**: 「書き方の原則」に照らし、決定の理由が書かれているか／冗長でないか／Non-Goals と Alternatives があるかを点検する。
5. **格納場所を決める**: リポジトリ内 `docs/design/` などにファイルで置くか確認する。
